## 一、技术架构详细设计

### 1.1 移动端技术选型

| 维度 | Flutter | React Native |
|------|---------|--------------|
| 渲染性能 | 自绘引擎，原生体验一致 | 原生组件，部分场景有桥接开销 |
| 开发效率 | 单代码库，双端体验一致 | 单代码库，但平台差异需额外适配 |
| 生态成熟度 | 非常成熟（2023+爆发） | 非常成熟（2015起步） |
| 团队技术栈 | Dart 学习成本 | JavaScript/TypeScript 更易招人 |
| AI能力集成 | plugin 生态丰富，FFI 调用方便 | bridge 模式调用 AI SDK 更成熟 |
| 视频处理 | `video_player`/`FFmpeg Kit` 支持好 | `react-native-video` 成熟但视频剪辑库少 |
| 推荐结论 | **首选 Flutter**（性能一致、开发效率高） | 备选 RN（招人容易、生态广） |

**推荐：Flutter 3.x**
- 理由：视频处理对性能敏感，Flutter 自绘引擎能保证抖音级流畅体验
- 抖快/视频号等国内 App 大量采用 Flutter
- 建议配合 `ffmpeg_kit_flutter` 做端上预处理

### 1.2 后端架构

#### 方案对比

| 维度 | Spring Boot（Java/Kotlin） | FastAPI（Python） |
|------|---------------------------|-------------------|
| 生态成熟度 | ★★★★★（企业级） | ★★★★（增长快） |
| AI/ML 集成 | 一般（需 pyjnius/pybind） | **首选**（原生 Python） |
| 视频/媒体处理 | 可用，但非主战场 | 可用，ffmpeg 命令封装方便 |
| 并发模型 | 线程池，同步阻塞友好 | asyncio + uvloop，高并发低成本 |
| 部署复杂度 | 高（JVM） | 低（单进程/Gunicorn） |
| 微服务治理 | 完善（Spring Cloud） | 需自建或借助 K8s/Istio |
| 推荐结论 | **适合纯企业级/强监管场景** | **首选 FastAPI**（AI 集成优势明显） |

**推荐：FastAPI + Python 3.11+**

理由：
1. **AI 能力集成成本最低** — 脚本生成、TTS、数字人 SDK 几乎全是 Python 接口
2. **异步架构** — 视频任务多为 IO 密集型，asyncio 吞吐率高
3. **前后端统一语言** — Flutter + FastAPI 都是现代声明式风格
4. **MVP 阶段效率最高** — 快速迭代，无需复杂微服务治理

**架构演进路径：**

```
MVP阶段（0→1万MAU）
  单体 FastAPI + PostgreSQL + Redis + MinIO + Kafka（单节点）
       ↓
成长期（1万→10万MAU）
  FastAPI 微服务拆分（用户服务/生产服务/分发服务）
  Kubernetes + ArgoCD 部署
       ↓
规模期（10万→100万MAU）
  全面微服务化，Service Mesh（Istio）
  多地域 CDN + 多活数据库
```

### 1.4 视频处理流水线（FFmpeg 方案）

#### 整体流程

```
原始素材上传
    ↓
[阶段1] 素材预处理（转码统一格式，提取元数据）
    ↓
[阶段2] AI 处理（字幕生成 / 语音识别）
    ↓
[阶段3] 混剪合成（多段拼接 + 转场 + BGM）
    ↓
[阶段4] 渲染导出（最终编码）
    ↓
[阶段5] 质量检查（视频完整性 + 封面生成）
    ↓
输出成品 → CDN 分发
```

#### FFmpeg 核心命令模板

```bash
# 素材预处理：统一转码为 H.264/AAC MP4
ffmpeg -i input.mov -c:v libx264 -preset fast -crf 23 \
  -c:a aac -b:a 128k -movflags +faststart \
  -vf "scale=1080:1920:force_original_aspect_ratio=decrease,pad=1080:1920:(ow-iw)/2:(oh-ih)/2" \
  output_1080p.mp4

# 混剪合成：多段拼接（concat）
ffmpeg -f concat -safe 0 -i filelist.txt -c copy merged.mp4

# 转场混剪：叠化转场（crossfade）
ffmpeg -i clip1.mp4 -i clip2.mp4 -filter_complex \
  "[0:v][1:v]xfade=transition=fade:duration=0.5:offset=5[outv]" \
  -map "[outv]" -c:v libx264 -preset fast merged.mp4

# 字幕烧录（ass/srt）
ffmpeg -i video.mp4 -vf "ass=subtitle.ass" -c:a copy output_with_sub.mp4

# 最终导出（多平台规格）
## 抖音/快手：9:16 1080x1920 H.264
## 视频号：16:9 1080x1080 H.264
## 小红书：支持多比例
```

#### 流水线技术架构

```
任务入队（Kafka / Redis Streams）
    ↓
Worker Pod（N台，每台部署 FFmpeg + Python worker）
    ├── 任务状态：PENDING → PROCESSING → COMPLETED / FAILED
    ├── 断点续传：任务元数据持久化 PostgreSQL
    └── 资源隔离：cgroup 限制 CPU/内存/磁盘 IO
    ↓
结果回写（CDN 临时存储 → OSS 最终存储）
    ↓
回调通知（WebSocket / SSE 给客户端）
```

**关键设计：**
- **依赖管理**：FFmpeg 版本必须固定（推荐 `ffmpeg:6.1-full`，不要用最新稳定版）
- **并发控制**：单 Worker 同时只处理 1 个视频，通过 K8s HPA 按 Kafka lag 扩容
- **磁盘清理**：处理完成后立即删除原始文件，CDN 缓存 7 天后自动失效
- **规格管理**：预定义各平台输出规格（分辨率/码率/帧率），存库可配置

### 1.5 数据库选型

#### 主数据库

| 场景 | 推荐方案 | 说明 |
|------|---------|------|
| 业务主库（用户/订单/任务） | **PostgreSQL 16** | 强事务，JSONB 灵活扩展，PostGIS 备选 |
| 任务队列状态 | **PostgreSQL + LISTEN/NOTIFY** | 轻量级，不引入额外组件 |
| 配置/模板/策略数据 | **PostgreSQL JSONB** | 免 MongoDB，一个库搞定 |
| 视频/脚本素材元数据 | **PostgreSQL** | 大量结构化字段，JSONB 存扩展属性 |
| 缓存层 | **Redis 7.x（Cluster 模式）** | session、限流、热点数据、AI 结果缓存 |
| 时序/监控数据 | **VictoriaMetrics / Prometheus** | 视频处理耗时、队列深度、CDN 命中率 |
| 日志 | **Loki + Grafana** | 结构化日志聚合 |
| **结论** | **PostgreSQL + Redis 足够**，不必引入 MongoDB | |

**不用 MongoDB 的理由：**
- PostgreSQL 16 JSONB 已经能存半结构化数据
- 团队已有 PostgreSQL 能力，无额外学习成本
- 视频元数据本质是强 schema，不适合文档数据库的随意性

### 1.6 存储方案（OSS + CDN）

#### 分层存储架构

```
本地 SSD（NVMe）
  └── 临时文件：正在处理的视频分片、FFmpeg 读写缓存
      淘汰策略：处理完成立即删除，保留时长 < 30min

对象存储（OSS）
  ├── 原始素材上传桶（内网上行，免费）
  ├── 成品视频存储桶（CDN 回源层）
  ├── 封面/缩略图桶（独立小文件，高访问）
  └── 备份桶（跨区域复制，重要数据）
      生命周期：素材7天后转归档，成品永久保留（除非用户删除）

CDN（分发层）
  ├── 视频 CDN：抖音/快手同款方案
  │   ├── 阿里云 CDN（全球节点，最成熟）
  │   ├── 腾讯云 ECDN（国内性价比高）
  │   └── 火山引擎 CDN（字节系，便宜）
  └── 静态 CDN：封面/JS/CSS
      └── 推荐：Cloudflare（免费套餐强大）或阿里云 OSS+DCDN
```

**CDN 推荐：**
- **国内首选：火山引擎 CDN** — 字节跳动嫡系，与抖音/今日头条同网，价格比阿里低 20~30%
- **备选：阿里云 CDN** — 节点最广，稳定性最好
- **出海：Cloudflare** — 全球 Anycast，免费 DDoS 防护

**CDN 缓存策略：**
```
视频文件（mp4）：Cache-Control: public, max-age=2592000（30天）
封面图（jpg）：Cache-Control: public, max-age=604800（7天）
API 响应：Cache-Control: private, no-cache
```

### 1.7 消息队列选型

| 维度 | Apache Kafka | Redis Streams | RabbitMQ |
|------|-------------|---------------|----------|
| 吞吐量 | 百万级/秒 | 万级/秒 | 万级/秒 |
| 消息持久化 | 强 | 中等 | 中等 |
| 消费模式 | 发布/订阅 + 消费组 | 发布/订阅 + consumer group | 多种 exchange 类型 |
| 运维复杂度 | 高（Zookeeper/KRaft） | 低（复用 Redis） | 中等 |
| 视频任务队列 | **首选**（积压能力强） | 备选（MVP阶段轻量） | 不推荐 |
| 推荐结论 | **Kafka（生产级）/ Redis Streams（MVP）** | 备选 | 不推荐 |

**MVP 阶段推荐：Redis Streams**
- 理由：无需额外部署 Kafka，Redis 已有，运维简单
- 消费组支持，重试/死信机制完备
- 缺点：单节点 Redis 性能上限 ~10万/秒，够用到 10万 MAU

**规模化阶段迁移到 Kafka：**
- Redis Streams → Kafka 迁移成本低（接口层抽象）
- Kafka 按 topic 拆分：视频处理任务 / 分发通知 / AI 回调

### 2.2 智能混剪引擎模块设计

#### 核心能力

1. **素材理解** — 镜头分割、场景识别、情绪标签
2. **智能剪辑** — 按脚本时间线自动拼接 + 转场
3. **元素叠加** — 字幕/BGM/特效/水印
4. **批量生产** — 一个脚本 × N 套素材 = N 个视频

#### 混剪算法流程

```
Step 1: 素材分析（FFprobe + OpenCV）
  输入：原始视频列表
  处理：
    - 镜头边界检测（场景切换检测）
    - 提取片段：start_time, end_time, duration
    - 打标签：场景类型（室内/户外/特写/全景）
    - 提取音频特征（响度/语速/情绪）
  输出：片段库 { clips: [{ id, path, start, end, tags, audio_level }] }

Step 2: 脚本解析
  输入：AI 生成的脚本（带时间戳）
  处理：
    - 提取每个时间点需要的画面类型
    - 计算所需片段数量和总时长
  输出：剪辑清单 { segments: [{ start, end, required_tags }] }

Step 3: 片段匹配（贪心算法 + 随机扰动）
  输入：剪辑清单 + 片段库
  处理：
    - 按 required_tags 过滤可用片段
    - 贪心选择最匹配片段
    - 添加随机扰动（同场景随机替换）避免机械化
  输出：选定片段序列

Step 4: 合成渲染（FFmpeg）
  输入：选定片段序列 + 字幕 + BGM + 转场
  处理：
    - 逐段拼接 + 字幕烧录 + 音频混音
    - 输出最终 MP4
  输出：成品视频文件
```

#### 防止"机械化"策略

```
问题：AI 批量生成的视频容易千篇一律，被平台降权
解决：
1. 片段随机化：同一脚本每次生成随机选不同片段
2. 顺序随机化：同类型片段打乱顺序
3. 字幕样式随机化：字体颜色/位置/动画随机
4. BGM 随机：同一情绪标签下随机选曲
5. 转场类型：随机选择 fade/slide/zoom/dissolve
6. 封面生成：每次随机不同排版
```

### 2.4 A/B 测试系统设计

#### 测试类型

1. **脚本 A/B 测试** — 同一素材测试不同脚本效果
2. **剪辑风格测试** — 同一脚本测试不同剪辑节奏
3. **分发时间测试** — 不同时段发布测试流量差异
4. **封面测试** — 同一视频不同封面点击率对比

#### 数据模型

```
Table: ab_experiments
  id, name, description, hypothesis,
  traffic_split_json,  // {"control": 0.5, "variant_a": 0.3, "variant_b": 0.2}
  start_time, end_time,
  status: draft/running/paused/finished

Table: ab_variants
  id, experiment_id, variant_key,  // control / variant_a / variant_b
  config_json,  // 具体参数
  sample_size, conversions, views

Table: ab_assignments（用户分配记录）
  id, experiment_id, user_id, assigned_variant,
  created_at  // 同一用户同 experiment 命中同 variant（一致性哈希）
```

#### A/B 分流算法

```python
def assign_variant(user_id: str, experiment_id: str, traffic_split: dict) -> str:
    """一致性哈希：同一用户永远命中同一 variant"""
    import hashlib
    key = f"{user_id}:{experiment_id}"
    hash_val = int(hashlib.md5(key.encode()).hexdigest(), 16)
    total = sum(traffic_split.values())
    cumulative = 0
    for variant, ratio in traffic_split.items():
        cumulative += ratio * 100  # 转为百分比
        if hash_val % 100 < cumulative:
            return variant
    return list(traffic_split.keys())[0]
```

#### 效果追踪

```
视频发布后自动创建 A/B 测试追踪任务：
  追踪指标（按优先级）：
    1. 播放完成率（3秒留存/完播率）← 核心指标
    2. 点赞/评论/分享转化率
    3. 关注转化率
    4. 点击率（CTR，封面决定）
    5. 主页访问率

数据源：
  - 各平台 API 回推（抖音/快手提供数据回调）
  - 定期拉取（平台 API 通常有 24h 延迟）
```

```
- 月活用户：每天平均生产 2 条视频
- 视频平均时长：60 秒
- 视频平均大小：H.264 1080p ≈ 15MB/分钟
```

### 4.2 自研 vs 采购对比

| 能力 | 自研成本（月） | 采购方案 | 推荐 |
|------|-------------|---------|------|
| 视频剪辑引擎 | ¥5万+（FFmpeg 研发） | 有赞/微盟 SaaS | **初期采购，规模化自研** |
| AI 脚本生成 | ¥0（API 成本） | 采购意义不大 | **自研（API 成本低）** |
| TTS 能力 | ¥0（API 成本） | 私有化 TTS 引擎 | **API 采购（自研 TTS 不划算）** |
| 数字人 | 无法自研 | 腾讯/阿里按分钟计费 | **初期采购，量大考虑定制** |
| 多平台分发 | ¥3万+（各平台 API 对接） | 新榜/神策等 SaaS | **采购为主，自研补充** |
| 视频 CDN | ¥0（云服务商提供） | 自建 CDN 不现实 | **采购云服务** |
| A/B 测试平台 | ¥1万+（数据采集分析） | Firebase/Optimizely | **采购（Firebase 免费够用）** |

**核心结论：**
1. **不要自研视频处理底层** — FFmpeg 社区强大，用好开源工具链即可
2. **AI 能力全部走 API** — 自研大模型成本极高，不适合创业公司
3. **采购优先，快速验证** — SaaS 采购 > 自研，除非你有独特算法壁垒
4. **规模化后考虑降本** — 月活 10 万+ 后，GPT-4o 替换 GPT-4o-mini，TTS 谈企业折扣

```

  - 一键分发

  - A/B 测试系统
  - 数据看板（播放量/转化率）
  - 封面生成器

Phase 5（13周+）：规模化
  - 数字人集成
```
